Introduction

Universal Blob Storage Adapter (UBSA)

UBSA is a Java library that provides a unified async client for object storage services including S3, Azure Blob Storage, and Google Cloud Storage.

It offers a single, consistent API across providers, making it easy to switch backends without rewriting your application code.

Why UBSA?

Migrating object storage across providers is painful:

  • Different SDKs and APIs
  • Inconsistent exception handling
  • Vendor lock-in

UBSA solves this with a two-step approach:

  1. Migrate in increments: incrementally rewrite your storage code to use the UBSA API
  2. Switch anytime: change the underlying provider via configuration

This means that after the initial migration, switching between S3, Azure Blob Storage, or GCS no longer requires code changes—only passing a different client to the factory.

Features

  • Multi-provider support (S3, Azure Blob Storage, GCS, S3-compatible)
  • Native async and sync clients
  • Built on top of official SDKs
  • Simple, consistent API
  • Streaming support, pre-signed URLs, and access to native clients

Example

S3Client nativeClient;
// BlobServiceAsyncClient nativeClient;

BlobStorageAsyncClient client = BlobStorageClientFactory.getAsyncClient(nativeClient);

// Same API regardless of provider
client.createBucket(...);
client.createBlob(...);
client.getBlob(...);

Overview

Universal Blob Storage Adapter (UBSA) is a universal object storage client that can access S3, Azure Blob Storage and Google Cloud Storage using a unified interface. It uses their Java SDKs under the hood, so configuration and behaviour is the same. UBSA has both an asynchronous and synchronous client. It is first, or one of the first libraries to ever make an async nonblocking universal client for blob storage.

UBSA keeps client configuration native, but storage operations and exception handling unified. And because it uses SDKs under the hood, it allows you to call methods that are outside the interface too, with the configuration of the client you provided. This again makes it the first solution of its kind to provide forward compatibility.

It is a wrapper around S3Client, BlobServiceClient, Storage etc. You call it via its unified interface, it then calls these SDK clients. Ubsa is a Java (JVM ecosystem) library and all that is needed to install it is to download it from Maven Central.

UBSA is a tool that can help when migrating Object Storage providers, or even just Object Storage SDKs with the same provider (GCP). You can incrementally rewrite your storage operations to use the UBSA client. When you are done, you switch the configuration provided to UBSA, and it will start using a new SDK under the hood.

Link to Maven Central: https://central.sonatype.com/artifact/io.github.michael-cirkl/ubsa/overview

Getting Started

This page describes how to use the UBSA library.

This shows a minimal example how to set up the library.

  1. Download the UBSA library from Maven Central https://central.sonatype.com/artifact/io.github.michael-cirkl/ubsa/overview
  2. Download a Java Blob Storage SDK, either for S3, Azure Blob Storage or Google Cloud Storage.
  3. Configure a client from the installed SDK (S3Client, S3AsyncClient, BlobServiceClient, BlobServiceAsyncClient, Storage)

For example:

S3Client s3 = S3Client.builder()
    .region(Region.US_EAST_1)
    .credentialsProvider(DefaultCredentialsProvider.create())
    .build();
  1. Create a universal client from the library using the provided BlobStorageClientFactory. Example for S3:
// Sync
BlobStorageSyncClient client = BlobStorageClientFactory.getSyncClient(s3);
// Async
BlobStorageAsyncClient client = BlobStorageClientFactory.getAsyncClient(s3);
  1. Now you can call storage operations on your universal client.
client.createBucket(...);
client.createBlob(...);
client.getBlob(...);

And that's it. You are welcome to build any abstractions on top of it, the library accepts any configured client, so if you are using Dependency Injection, S3 Quarkus extension or anything similar, that is supported too.

If you want async Google Cloud Storage Client, create Storage with .grpc() and pass it into createAsyncClient().

A recommended way to use Ubsa for migrations is making a wrapper class with references to both the native client and the library client, and incrementally rewriting your application to use the library's client. This allows more careful testing and can avoid unforeseen issues.

Calling Unsupported Methods Natively

The library provides a synchronous and asynchronous unified client for accessing blob storage. However, it does not support all possible storage methods. The library allows you to unwrap a native configured client (S3Client, BlobServiceClient, Storage etc.) from the library's unified client, and use it to call a method that is only available on for example S3 SDK.

BlobStorageSyncClient client = ...
S3Client nativeClient = client.unwrap(S3Client.class);
nativeClient.createMultipartUpload(...);

While doing this in methods is possible, a more maintainable way to do this is to create a wrapper class, that holds references to both the clients, implements the interface and adds extra methods. For example:

class ClientWrapper implements BlobStorageSyncClient {
   S3Client nativeClient;
   BlobStorageSyncClient client;

   @Override
   Boolean bucketExists(String bucketName) {
       return client.bucketExists(bucketName);
   }

   CreateMultipartUploadResponse createMultipartUpload(CreateMultipartUploadRequest request) {
      return nativeClient.createMultipartUpload(request);
   }

   .... other methods
}

This is an S3 example but it is possible with all 3 supported SDKs. With both sync and async. Since the library's client is created by passing the native SDK's client into BlobStorageClientFactory, unwrapping is not strictly necessary here, as you can just configure lets say S3Client, then pass it into BlobStorageClientFactory.getSyncClient(nativeClient), which will return BlobStorageSyncClient.

Exception Handling

The library emits two types of exceptions. UbsaException for errors related to storage, and IllegalArgumentException for basic validations (negative number of seconds etc.)

UbsaException is a replacement for S3Exception, BlobStorageException and StorageException from S3, Azure and GCP SDKs. Just like them, it contains a method to get the HTTP status code of the response. This is taken from the original exceptions from the SDK, but it can be null if the cause of the error is an IOException, where a status code is not available.

When using S3 or Azure Blob Storage, the library throws more specific exceptions, such as BucketNotFoundException or AccessDeniedException. These are subtypes of UbsaException, meaning that all storage exceptions will be caught, if you catch UbsaException. Google Cloud Storage SDK does not expose the cause of the error, so when using GCS SDK for the client, the library only throws UbsaException.

This means that if you are migrating from S3/Azure to Google Cloud Storage, you will need to rewrite your exception handling to only catch UbsaException. However, if you are for example currently using S3 and want to migrate to Azure, you can keep catching specific exceptions, and just rewrite them from specific S3 exceptions to specific Ubsa exceptions.

The library contains these specific exception types. Not all S3/Azure exception causes are covered, the ones without a mapping throw UbsaException. If you are interested to see which S3 and Azure Blob Storage exception causes map to these exception types, check client.exception.types package, or read it from a javadoc report, since this information is contained in javadoc comments.

  • AccessDeniedException
  • AuthenticationFailedException
  • BlobNotFoundException
  • BucketNotFoundException
  • BucketAlreadyExistsException
  • ConditionFailedException
  • InvalidRequestException
  • RequestTimeoutException

Local Build and Testing

Build

This section describes how to build and run tests, if you want to make changes to the library locally.

The library is a multi-module maven project. The code of the library is in the blob-storage-adapter module, and tests are in the testing module. This allows you to test the library with different version SDKs than it was built with, which is supported behaviour.

Building is quite simple, the library itself (blob-storage-adapter module) can be built with maven package

Test

The library includes several integration tests and testing scenarios that cover basically all methods of the client interfaces. They are located in the testing module. The test environment is configured via a .env file you should place in the root of the main project. There is a .env.example file that shows possible configuration. The .env file is used to provide credentials that are going to be used by the SDKs when testing.

Azure Blob Storage and S3 have emulators of their blob storage which can be used for local testing. It is much faster compared to making network calls to real datacenters, and free, therefore it is recommended to use them. The .env.example file as is included in the repository already has the right credentials to connect and use these emulators. All you need is to launch them, which can be done via docker-compose.yml file in root of the project.

If you don't configure credentials for a particular SDK in the .env file, the tests for that SDK will be skipped. Google cloud does not have a working emulator, so if you copy the .env.example file to .env 1:1, it will launch S3 and Azure Blob Storage tests on emulators, and skip Google Cloud tests.

Google Cloud setup

If you want to run even the Google Cloud tests, you need to create a Google Cloud account and run the tests on their public cloud infrastructure. They do have a 90-day limited free tier.

gcloud iam service-accounts keys create key.json --iam-account=my-service-account@YOUR_PROJECT_ID.iam.gserviceaccount.com

  1. Setup Google Cloud account on https://console.cloud.google.com/
  2. Create a project. Put the id of the project into the .env file as shown in the example
  3. Create a service account (Service accounts tab under IAM & Admin)
  4. Create a custom role (Roles tab accounts tab under IAM & Admin)
  5. Give the role the needed permissions (last section of this wiki page)
  6. Assign the role to the service account (Service accounts -> Permissions -> Manage Access -> Assign roles -> Add another role)
  7. Download gcloud CLI from https://cloud.google.com/cli
  8. Login to the CLI using gcloud auth login
  9. Set your project id using gcloud config set project YOUR_PROJECT_ID
  10. Open the detail of the service account you created in Google Cloud Console. Copy the email. Then generate a key locally, and use the email as the parameter here, instead of the example:
  11. Run the tests with this environment variable pointing to that key file: GOOGLE_APPLICATION_CREDENTIALS=absolutePathToTheKey

If it is skipping GCP tests with error: Skipping GCP tests: no GCP backend configured., you haven't put the project id in the .env file, as is shown in the example.

If it is skipping GCP tests with error: Skipping GCP tests: Application Default Credentials are not configured., you haven't set the GOOGLE_APPLICATION_CREDENTIALS environment variable.

While it is possible to login just with your Google Account and not via Service Account, it does not allow presigning URLs, so some tests will fail on permissions. Therefore it is reccomended to use a service account as shown above.
